1. PROJECT OVERVIEW
================================================================================

Dx is a full-featured, web-based Clinical Laboratory Information System
(LIS) designed for hospital and diagnostic laboratory use. It covers the complete
specimen lifecycle from patient registration and accessioning through to result
reporting, billing, and regulatory compliance.

The system was initially generated by an AI tool called "Antigravity" and has
been iteratively developed and maintained using Claude Code (Anthropic). It has
gone through ~40 versions between December 2025 and its current state.

The application name in-product is "Dx". The system title in the browser
tab is "Dx LIS".

Core capabilities:
  - Patient registration and MRN management
  - Test ordering and accession number generation
  - Specimen receiving, tracking (Chain of Custody), and storage
  - Phlebotomy scheduling and mobile collection workflows
  - Result entry, two-tier validation (Technical → Clinical)
  - Clinical intelligence: delta checks, critical values, reflex testing
  - Quality control (Westgard rules, Levey-Jennings charts)
  - Histopathology and Microbiology specialty modules
  - Billing and invoice generation
  - Reagent inventory management
  - Equipment maintenance logging
  - FHIR R4 interoperability facade
  - HL7/ASTM instrument middleware
  - Audit logging on all significant actions
  - Role-based access control
  - Configurable theming (colour + dark mode)

3. GETTING STARTED (LOCAL SETUP)
================================================================================

Prerequisites:
  - Node.js >= 20
  - npm >= 9

Installation:
  1. Clone the repository
  2. cd clinical-lis
  3. npm install
  4. npm run dev
     → Opens on http://localhost:3000

The database file (sqlite_v2.db) is checked in to the repository and already
contains seed data. You do NOT need to run migrations from scratch unless you
are setting up a blank environment.

To reset the database from scratch:
  npx tsx src/scripts/reset-db.ts
  npx tsx src/scripts/seed-from-json.ts   (optional: loads legacy JSON data)

To run database migrations (after schema changes):
  npx drizzle-kit generate    # generates migration SQL in drizzle/
  npx drizzle-kit migrate     # applies pending migrations

To build for production:
  npm run build
  npm run start

The instrument server (separate service) must be started independently:
  cd instrument-server
  npm install
  npm start
  → Listens on TCP port (configured inside server.ts)

Environment variables (.env file — not committed):
  No required env vars for basic operation. The DB path is hardcoded as
  'sqlite_v2.db' in src/db/index.ts. For production, consider:
    DATABASE_URL=./sqlite_v2.db
    SMTP_HOST=...
    SMTP_PORT=...
    SMTP_USER=...
    SMTP_PASS=...
    NEXTAUTH_SECRET=...  (not currently used — custom auth)

5b. DUAL DB ACCESS PATTERN — CRITICAL
---------------------------------------

There are TWO ways the codebase talks to the database. Understanding this is
essential for maintaining consistency.

PATTERN A — Modern (Drizzle ORM, PREFERRED):
  import { db } from '@/db';
  import { results, orders } from '@/db/schema';
  import { eq } from 'drizzle-orm';

  const rows = await db.select().from(results).where(eq(results.orderId, id));
  await db.insert(results).values({ id: uuid(), ... });
  await db.update(orders).set({ status: 'Completed' }).where(eq(orders.id, id));

  Used by: auth, results, orders, clinical engine, delta checks, critical values,
           reflex rules, most newer API routes.

PATTERN B — Legacy JSON Adapter (AVOID for new code):
  import { readDb, writeDb } from '@/lib/db';

  const db = await readDb();   // loads ENTIRE database into a JS object
  db.orders.push(newOrder);    // modify in memory
  await writeDb(db);           // upsert everything back

  Used by: FHIR facade, middleware ingest, some older admin routes.

  WHY IT EXISTS: The original system used a JSON file as the database. When
  migrated to SQLite, readDb/writeDb was added as a compatibility shim so the
  existing route handlers didn't need to be rewritten all at once.

  WHY IT'S PROBLEMATIC:
    - readDb() runs ~35 parallel SELECT * queries on every call
    - No transactional safety — two concurrent requests can overwrite each other
    - The in-memory object is a snapshot; stale between reads
    - writeDb() loops through arrays doing individual upserts — O(n) per table

  ACTION REQUIRED: When you touch any route that still uses readDb/writeDb,
  migrate it to direct Drizzle queries. Do not write new code using this pattern.


5c. VIRTUAL TABLES (SETTINGS-BACKED)
--------------------------------------

Some collections that should be proper tables are stored as JSON strings inside
the 'settings' table. They were added before the schema was fully normalised.

  settings.key            | Used as
  ------------------------|------------------------------------------
  'specimenTypes'         | Array of specimen type objects
  'commentCodes'          | Array of pre-defined comment templates
  'manualSections'        | Lab manual section data
  'storageLocations'      | Storage unit/rack/box definitions
  'cocEvents'             | Chain of Custody event log (all orders)
  'orderPayments'         | Payment status per order

These are read back by readDb() and returned as top-level properties. API routes
that manage these write them back via writeDb() or directly update the settings
table with JSON.stringify().

Ideal future state: migrate each to its own proper table.

6. AUTHENTICATION & SESSIONS
================================================================================

Flow:
  1. User POSTs to /api/auth/login with { username, password }
  2. Server fetches user from DB, verifies password with bcrypt
  3. If the stored password is plain-text (legacy), it is bcrypt-upgraded automatically
  4. A UUID session token is inserted into the 'sessions' table (24hr expiry)
  5. Two cookies are set (both httpOnly, sameSite=strict):
       auth_token    — the UUID token (primary, DB-validated on every request)
       auth_session  — legacy JSON cookie { id, username, role, name, department }
                       (kept for routes that haven't migrated to getAuthUser())

Token validation (modern routes):
  import { getAuthUser } from '@/lib/auth';
  const user = await getAuthUser();   // returns AuthUser | null
  if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });

  getAuthUser() checks auth_token → queries sessions table → queries users table.
  Falls back to auth_session JSON cookie if auth_token is missing (backward compat).

Route-level protection (legacy pattern, still common):
  const session = cookieStore.get('auth_session');
  if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  const currentUser = JSON.parse(session.value);

Middleware-level protection (src/middleware.ts):
  All routes except /login and /api/auth/* require a valid auth_token cookie.
  Without it, page requests are redirected to /login; API requests get 401.

Session expiry:
  Sessions expire after 24 hours (stored as Unix timestamp in sessions.expiresAt).
  There is NO session refresh mechanism — users must log in again after 24 hours.
  Expired sessions are not cleaned up automatically — add a cron or maintenance
  task to DELETE FROM sessions WHERE expires_at < unixepoch() * 1000.

Password policy:
  bcrypt with cost factor 12. No minimum length is enforced server-side currently.

7. ROLE-BASED ACCESS CONTROL (RBAC)
================================================================================

Five roles are defined (defined in AuthContext.tsx and enforced in Sidebar.tsx
and individual API routes):

  Role        | Description
  ------------|-----------------------------------------------------------
  admin       | Full system access. All admin pages, all configuration.
  manager     | Lab management. Most admin pages, no user management.
  scientist   | Bench scientist. Accessioning, result entry, QC.
  medic       | Pathologist/physician. Can clinically verify results.
  clerk       | Reception/admin staff. Accessioning, basic tracking.

UI enforcement (Sidebar.tsx):
  const isAdmin   = role === 'admin';
  const isManager = ['admin', 'manager'].includes(role);
  const isLabStaff = ['admin', 'manager', 'scientist', 'medic'].includes(role);

  Navigation items are conditionally shown based on these flags. This is
  UI-only — it does NOT prevent direct URL access. API routes must enforce
  their own RBAC.

API enforcement (varies by route):
  Some routes check role explicitly:
    if (!['admin', 'manager'].includes(currentUser.role)) {
      return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
    }

  Other routes (especially older ones) do not check role at all — they only
  check that the user is authenticated. This is a known gap.

Result validation workflow RBAC:
  - Any lab staff can enter results (status = 'Resulted')
  - action=TECHNICAL_VALIDATE sets 'Technically Validated' — meant for scientists
  - action=CLINICAL_VERIFY sets 'Clinically Verified' — meant for medic/pathologist
  - The API does not strictly enforce who can do which action (UI enforces it)

10. KEY LIBRARIES & UTILITIES

10a. CLINICAL ENGINE — src/lib/clinical-engine.ts
---------------------------------------------------

The most important business logic file. Called automatically by the results
POST route whenever a new result is entered (not on re-validation).

Functions:

  runDeltaChecks(orderId, patientId, testId, currentValue)
    - Finds previous results for the same patient+test (up to 5 previous orders)
    - Evaluates each enabled delta_check_rule against the change
    - Inserts a delta_check_flag row if a threshold is exceeded
    - Returns { flagged: boolean, flags: DeltaFlag[] }

  checkCriticalValues(orderId, patientId, testId, testCode, value, enteredBy)
    - Reads referenceRange.panicLow / panicHigh from test_definitions
    - Creates a critical_value_notification row if outside panic range
    - Returns boolean (was critical)

  evaluateReflexRules(orderId, testId, value)
    - Finds enabled reflex_rules where triggerTestId matches
    - Evaluates { operator, value } condition
    - If triggered, inserts a reflex_activation and appends the new test to the
      order's testIds array
    - Returns string[] of newly added test IDs

  checkNotifiableConditions(orderId, patientId, testId)
    - Finds notifiable_conditions that reference this testId
    - Creates an epidemiology_notification if not already created for this order
    - Returns boolean

  getDemographicRefRange(testId, ageYears, gender, isPregnant, trimester)
    - Scores and returns the most specific demographic reference range
    - Used by result entry UI to show appropriate reference intervals

  computeCalculatedTests(formulaType, inputs, patientAge, patientGender)
    - Supported formulas: 'ldl_friedewald', 'anion_gap', 'a_g_ratio', 'egfr_ckd_epi'
    - Returns a number or null if inputs are insufficient
    - CKD-EPI 2021 creatinine equation (accepts µmol/L, converts internally)
    - LDL Friedewald only valid when TG <= 4.5 mmol/L

All functions are called with .catch() — failures are logged but do not block
the main result entry flow (fire-and-forget pattern).


10b. ROUTING ENGINE — src/lib/routing-engine.ts
------------------------------------------------

A class-based engine for routing test orders to workstations.

  class RoutingEngine
    constructor(workstations, routingRules)
    async routeTest(request: RoutingRequest): Promise<RoutingResult>
    async routeBatch(requests: RoutingRequest[]): Promise<RoutingResult[]>
    async rerouteTest(originalAssignment, request): Promise<RoutingResult>

The engine selects workstations based on:
  - Configured routing rules per test
  - Sample type compatibility
  - Workstation status (online/offline/busy)
  - For STAT (priority <= 3): lowest queue depth
  - For routine: lowest utilisation (currentTests / maxThroughput)

NOTE: The routing engine is fully implemented but the UI routing page is mostly
scaffolding. The engine is not automatically invoked on order creation — it is
available for future integration.


10c. SECURITY — src/lib/security.ts
-------------------------------------

AES-256-GCM encryption helpers used exclusively by the backup/restore system.

  deriveKey(password, salt): Promise<Buffer>
    - Uses scrypt to derive a 32-byte key from a password

  createEncryptionDetails(password)
    - Generates salt (16 bytes) + IV (16 bytes), creates cipher
    - Returns { cipher, salt, iv }

  createDecryptionDetails(password, salt, iv)
    - Recreates decipher from stored salt and IV

Backup file format: [salt:16][iv:16][authTag:16][encryptedData...]


10d. ACCESSION NUMBER GENERATOR — src/lib/accession.ts
--------------------------------------------------------

  generateNextAccessionNumber(): Promise<string>
    - Format: YYYY-MM-DD-XXXX (e.g., 2026-06-29-0001)
    - Queries the orders table for the latest accession number matching today's
      prefix, then increments the sequence
    - Resets to 0001 each day
    - NOT atomic — concurrent requests could generate duplicates.
      A DB-level sequence or transaction would be safer.


10e. INVENTORY TRACKING — src/lib/inventory.ts
-----------------------------------------------

  consumeReagent(testId, userId): Promise<void>
    - Called automatically by the results POST route on new result entry
    - Decrements inventory quantity by 1 for reagents linked to the test
    - Creates an inventory_transaction record
    - Logs a warning alert if quantity falls below minThreshold


10f. UTILITY FUNCTIONS — src/lib/utils.ts
------------------------------------------

  fuzzyMatch(query, target): boolean
    - Simple substring matching for patient/order search

  generateZPLLabel(accessionNumber, patientName, testCodes): string
    - Generates Zebra ZPL barcode label string for label printers

  evaluateResult(value, referenceRange): { status, flag }
    - Compares a numeric value against referenceRange { min, max, panicLow, panicHigh }
    - Returns status ('normal'|'high'|'low'|'critical') and flag ('H'|'L'|'HH'|'LL')
    - Used by middleware ingest for auto-flagging

12. THEMING SYSTEM
================================================================================

The system supports multiple colour themes and a dark mode.

Themes available: blue (default), emerald, rose
Modes: light (default), dark

How it works:
  1. Theme and mode are stored in the 'settings' table as:
       settings.key = 'themeColor', settings.value = '"blue"'
       settings.key = 'themeMode', settings.value = '"dark"'

  2. RootLayout (src/app/layout.tsx) reads these on every server render and
     applies classes to <body>:
       theme-blue (or theme-emerald / theme-rose)
       mode-dark  (if dark mode)

  3. globals.css defines CSS variables per theme class:
       :root { --primary-500: #3b82f6; }
       .theme-emerald { --primary-500: #10b981; }
       .theme-rose { --primary-500: #f43f5e; }
       .mode-dark { --slate-50: #0f172a; --slate-900: #f8fafc; }

  4. Tailwind uses these variables via tailwind.config.js:
       colors: { primary: { 500: 'var(--primary-500)', ... } }

  5. When user changes theme in /admin/settings, the page also applies
     the classes immediately client-side (without a page reload) via
     document.body.classList manipulation.

The sidebar always uses dark styling (#0f172a background) regardless of theme.
The theme only affects the main content area (primary colour accents).

13. INSTRUMENT SERVER (SEPARATE SERVICE)
================================================================================

Location: instrument-server/
Purpose: TCP socket server that communicates with physical lab analyzers
         using HL7 v2 or ASTM protocol.

How it works:
  1. Instrument sends results over TCP to this server's port
  2. Server parses the HL7/ASTM message
  3. Server POSTs the parsed results to the main LIS API:
       POST /api/middleware/ingest
       Body: { instrumentId, sampleId, timestamp, results: [...] }
  4. The LIS API finds the order, creates result records, updates status

The server is configured inside instrument-server/src/server.ts.
It uses axios for HTTP POSTs back to the LIS.

Running:
  cd instrument-server
  npm install
  npm start    (production)
  npm run dev  (development with hot reload via tsx)

Docker:
  instrument-server/Dockerfile — containerises the server
  docker-compose.yml — runs both the LIS app and the instrument server together

15. MIDDLEWARE INGEST API
================================================================================

Route: POST /api/middleware/ingest
Body: { instrumentId, sampleId, timestamp, results: [{ testCode, value, units, flags }] }

This endpoint simulates receiving parsed HL7 ORU messages from an interface

CRITICAL SECURITY ISSUE:
  The authentication check on this route is COMMENTED OUT (line 16):
    // if (authHeader !== 'Bearer MIDDLEWARE_SECRET_KEY') {
    //   return NextResponse.json({ error: 'Unauthorized Interface' }, { status: 401 });
    // }

  This means ANY HTTP client can POST results to this endpoint without any
  credentials. Fix immediately before any production deployment:
    1. Uncomment the auth check
    2. Store the secret key in an environment variable (not hardcoded)
    3. Configure the instrument-server to send the correct Authorization header

17. BACKUP & RESTORE
================================================================================

Route: POST /api/admin/backup   — Creates encrypted SQLite backup
Route: POST /api/admin/restore  — Restores from encrypted backup

Encryption: AES-256-GCM (see src/lib/security.ts)
Key derivation: scrypt from a user-provided password
File format: [salt:16][iv:16][authTag:16][encrypted SQLite bytes]

The backup creates a copy of sqlite_v2.db, encrypts it, and returns it as a
downloadable binary file. The restore process decrypts and replaces the active
database file.

CAUTION: The restore operation replaces the live database — it is destructive.
Always keep an external copy of backups.

18. KNOWN ISSUES & TECHNICAL DEBT
================================================================================

SECURITY (fix before production):
  1. Middleware ingest auth is disabled — src/app/api/middleware/ingest/route.ts:16
     Anyone can POST instrument results without credentials.

  2. FHIR endpoint has no authentication — patient data is publicly accessible
     to anyone who knows the endpoint URL.

  3. The legacy auth_session JSON cookie is trusted directly by some API routes
     without DB re-validation. A user who can write their own cookie (unlikely
     with httpOnly but worth noting) could spoof a role.

  4. Session cleanup — expired sessions are never purged from the sessions table.
     Add: DELETE FROM sessions WHERE expires_at < (unixepoch() * 1000)

DATA INTEGRITY:
  5. No UNIQUE(orderId, testId) constraint in results table — duplicates are
     prevented by application logic only (SELECT then upsert). The SELECT and
     INSERT are not in a transaction, creating a race window.

  6. Accession number generation (generateNextAccessionNumber) is not atomic.
     Two simultaneous accessioning requests could generate the same number.
     Fix: use a DB-level auto-increment or a transaction with a lock.

  7. testIds stored as JSON string in orders — prevents efficient JOINs and
     filtered queries. Should be normalised to an order_tests junction table.

PERFORMANCE:
  8. readDb() loads the entire database on every call. For a lab with thousands
     of orders, this will be slow. All routes using readDb/writeDb should be
     migrated to targeted Drizzle queries.

  9. The FHIR route uses readDb() — adding auth and migrating to Drizzle would
     dramatically improve response time.

COMPLETENESS:
  10. LOINC codes table exists but is not linked to test definitions or FHIR output.

  11. Calculated tests (eGFR, LDL, etc.) are computed by the engine but there is
      no automatic trigger to run them when the required input tests are resulted.

  12. The routing engine is fully functional but not automatically invoked when
      orders are created.

  13. No background job system — email queue, session cleanup, and TAT breach
      detection all require manual triggers or API calls to execute.

TESTING:
  14. Zero test coverage — no unit tests, integration tests, or end-to-end tests.
      Adding at minimum API integration tests (e.g., with Vitest + supertest or
      Playwright) is strongly recommended before extending the system.

HOUSEKEEPING:
  15. Root directory contains development artefacts that can be deleted:
        errors.txt, errors_v6.txt ... errors_v9.txt
        tsc_errors.txt, tsc_errors_v2.txt ... tsc_errors_v5.txt
        errors_final.txt, build_log.txt, build-errors.txt

  16. sqlite.db (the old v1 database) can be deleted — sqlite_v2.db is active.
